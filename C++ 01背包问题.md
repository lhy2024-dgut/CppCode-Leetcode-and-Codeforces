### 💻 架构师专属：01 背包一维 DP 通用模板 (C++)

```cpp
#include <iostream>
#include <vector>
#include <algorithm>

using namespace std;

// 通用 01 背包求解函数
// W: 背包的最大承重
// wt: 每个物品的重量数组
// val: 每个物品的价值数组
int knapsack(int W, const vector<int>& wt, const vector<int>& val) {
    int n = wt.size(); // 物品的总数量
    
    // 1. 构建 DP 状态账本 (一维滚动数组)
    // dp[j] 表示背包容量为 j 时，能装下的最大价值。
    // 初始状态全为 0，因为容量为 0 或者还没装物品时，价值都是 0。
    vector<int> dp(W + 1, 0);

    // 2. 核心业务双重循环
    // 外层循环：一件一件地遍历所有物品
    for (int i = 0; i < n; i++) {
        
        // 内层循环：遍历背包容量 (⚠️ 极其重要的避坑点：必须从大到小逆序遍历！)
        // 为什么要逆序？
        // 因为 dp[j] 的更新依赖于上一层 (也就是还没考虑当前物品 i) 的 dp[j - wt[i]]。
        // 如果正序遍历，前面的值会被提前覆盖，导致一件物品被重复装入多次（这就变成完全背包了）。
        for (int j = W; j >= wt[i]; j--) {
            
            // 3. 状态转移方程（打擂台机制）
            // 策略 A：不拿这件物品，保持原样 (dp[j])
            // 策略 B：拿这件物品，腾出 wt[i] 的空间加上当前价值 (dp[j - wt[i]] + val[i])
            dp[j] = max(dp[j], dp[j - wt[i]] + val[i]);
            
        }
    }

    // 4. 返回最终结果：当背包容量刚好为 W 时的最大价值
    return dp[W];
}

int main() {
    // 业务测试数据
    int W = 4; // 背包总容量
    vector<int> wt = {1, 3, 4}; // 物品重量
    vector<int> val = {15, 20, 30}; // 物品价值

    cout << "最大能装入的价值是: " << knapsack(W, wt, val) << "\n";
    
    return 0;
}

```

<br><br><br><br><br><br><br><br>
### 重要 ： 外循环从 0 到 n，内循环从 W 到 wt[i] （内循环由大到小）
<br><br><br><br><br><br><br><br>

### 🧠 核心架构解析（为什么这么写？）

1. **状态定义 `dp[j]`：** 这里的 `j` 就是一个容量卡尺。`dp[j]` 回答的问题是：“如果我现在的包只能装 `j` 千克的东西，我最多能拿走多少钱？”
2. **核心转移动作 `max(不拿, 拿)`：** 这就是将 DFS 里的分叉决策直接落实成了数学公式。
3. **生死存亡的“逆序遍历”：** 注意看内层循环 `for (int j = W; j >= wt[i]; j--)`。这是把二维数组压缩成一维数组的最精髓操作！逆序更新可以完美保证，当前物品在当前这轮循环中，绝对**只被放入一次**（严格遵守了 01 背包中“0”和“1”的业务约束）。

### 状态的滚动刷新

您说得完全正确：**在刚开局（处理第 0 件物品）的时候，它们确实全都是 0！**

但这本 `dp` 账本不是静态的，它是随着外层循环（遍历每一件物品）**不断覆盖、进化**的。等您处理到第 1 件、第 2 件物品时，等号右边的 `dp` 数组里，早就已经写满了**上一轮留下来的“历史最优成绩”**了！

为了让您彻底看透这个魔法，我们把这行代码拆解成大白话，然后拿咱们刚才的测试数据，直接给您做一次“内存级别的单步调试”！

### 🔍 1. 公式的大白话翻译

`dp[j] = max(dp[j], dp[j - wt[i]] + val[i]);`

* **等号左边的 `dp[j]`：** 今天（处理完当前物品 `i` 后）我们要写进账本的最新纪录。
* **等号右边的 `dp[j]`（策略 A：不拿）：** **历史记录**。如果不拿手里这件新物品 `i`，仅靠以前装的物品，容量为 `j` 时能达到的最大价值。
* **等号右边的 `dp[j - wt[i]] + val[i]`（策略 B：拿）：** **破局策略**。如果我非要拿手里这件物品 `i`，那我必须强行在包里腾出 `wt[i]` 的空间。所以，我去历史账本里查一下：**“以前容量只有 `j - wt[i]` 的时候，最多装了多少钱？”** 查到之后，再加上手里这件新物品的价值 `val[i]`。

最后，`max()` 函数就是一个无情的裁判：策略 A 和 策略 B 谁赚得多，就把谁写进今天的账本！

* **当容量 $j=4$：核心高潮来了！**我要不要装这件重量 3、价值 20 的物品？
* **策略 A（不装）：** 查历史记录 `dp[4]`。看上面的账本，是 **15**。（意味着我只用之前的物品，容量 4 能赚 15）。
* **策略 B（装）：** 我得给它腾出 3 的空间。去查历史记录 `dp[4 - 3]` 也就是 `dp[1]`。看上面的账本，`dp[1]` 是 **15**！这意味着剩下的 1 容量，在以前的最优解里能赚 15。再加上手里这件新物品的 20，总共是 `15 + 20 = 35`！

* 更新账本：`dp[4] = 35`。


* 当容量 $j=3$：
* 不装：历史记录 `dp[3]` 是 15。
* 装：查历史记录 `dp[3 - 3]` 也就是 `dp[0]`，是 0。加上新价值 20，等于 20。

### 🧠 架构师深度延展：为什么不是翻倍？

既然您观察到了这么细微的层面，我必须借您这个完美的切入点，给您透漏一个动态规划里极其精妙的**“底层机制”**。

您试想一下，假设背包总容量是 10，第一件物品的重量 `wt[0]` 是 3，价值 `val[0]` 是 15。

咱们是**逆序（从后往前）**遍历的：

1. 算 `dp[10]` 时，它去查 `dp[10 - 3]` 也就是 `dp[7]`。因为是从后往前算，`dp[7]` 此时还没轮到更新，它还是初始的 **0**。所以 `dp[10] = 0 + 15 = 15`。
2. 算 `dp[9]` 时，查 `dp[6]`，也是 **0**。所以 `dp[9] = 15`。
3. 一路算到 `dp[6]`，查 `dp[3]`，还是 **0**。所以 `dp[6] = 15`。

**现在，刺激的来了！如果咱们把代码写错，写成了【正序（从前往后）】遍历呢？**

1. 先算 `dp[3]`：查 `dp[0]` (是 0)，`dp[3] = 0 + 15 = 15`。
2. 再算 `dp[6]`：查 `dp[6 - 3]` 也就是 `dp[3]`。**注意！此时 `dp[3]` 刚刚被更新成了 15！** 那么 `dp[6]` 就会变成 `15 + 15 = 30`！
3. 等算到 `dp[9]` 时，查 `dp[6]`，就会变成 `30 + 15 = 45`！

发现了吗？如果正序遍历，第一件物品就会被**无限次地重复装入**！这在业务上，就从“每种物品只有一个”的**01背包问题**，瞬间变成了“每种物品有无限个”的**完全背包问题**！
